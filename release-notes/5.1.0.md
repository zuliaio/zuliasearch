## New Features

- **More-like-this (MLT) queries.** A new `MORE_LIKE_THIS` query type finds documents similar to one or more sources, with three modes: lexical (Lucene MoreLikeThis term selection), vector (KNN search on the averaged source vectors, normalized for unit-vector fields), and hybrid (both combined). Sources can be document IDs (resolved server-side with a batch fetch on the federating node), raw text, and/or raw vectors, and at least one source is required. Each shard builds its lexical MLT query against its own IndexReader, so IDF statistics are per-shard accurate. Source documents are excluded from results by default (`includeSourceDocs` to opt in), `minimumShouldMatch` is honored on the per-field term disjunction, realtime searches see just-stored source documents, and wildcard index patterns and aliases resolve the same way as regular queries. Source documents are capped at 100 per query. Available in the Java client via the `MoreLikeThisQuery` builder (`forText` / `forDocuments` factories) and over REST through the existing `queryJson` parameter.

## Improvements

- **Queries with an empty index list now fail fast.** All query types now return a clear "Query requires at least one index" error instead of silently returning zero hits when the request names no indexes. Index-warming searches are unaffected since they imply their index.

## Bug Fixes

- **Associated document metadata was corrupted in single-node (filesystem) mode.** `FileDocumentStorage` wrote the protobuf debug string of the metadata (`ByteString.toString()`) instead of the metadata itself, so associated documents stored with metadata could not round-trip it. Metadata is now stored and fetched correctly, covered by new store/fetch round-trip tests.
- **Fixed a possible `ClassCastException` reading associated document timestamps in single-node mode.** `Document.parse` returns an `Integer` for whole numbers that fit in an int, so the unconditional cast to `long` could throw. The timestamp is now read as a `Number`.
