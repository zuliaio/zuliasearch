## New Features

- **DATE fields accept String and epoch-millisecond values.** A `DATE` field can now be stored from a String or a numeric epoch-milliseconds value, in addition to `java.util.Date`. Supported String forms: a full timestamp with an optional zone offset (`2024-06-17T16:10:00Z`, or offset-less `2024-06-17T16:10:00` assumed UTC), date (`2024-06-17`), year-month (`2024-06`), or year (`2024`); `-` and `/` separators are both accepted. `null` and blank values are treated as no value and skipped. The same conversion applies consistently across indexing, sorting, and faceting (flat and hierarchical).
- **Offset-less timestamps in date queries.** Date range queries now accept an offset-less timestamp bound (e.g. `published:[2024-06-17T16:10:00 TO ...]`), assumed UTC, matching the explicit `...Z` form. A standalone timestamp point query must be quoted (`published:"2024-06-17T16:10:00Z"`) or have its colons backslash-escaped, since `:` is otherwise read as a field separator.

## Improvements
- **Facet counts as a map (client).** `SearchResult.getFacetCountsAsMap(field)` and `getFacetCountsAsMap(field, paths...)` return a field's facet counts as a `Map<String, Long>` of value to count preserving the returned count-descending order 
- **Range query-string accessors (client).** `RangeFilter` now exposes `getBaseQuery()` (the range query string) and `toQueryString()` (the same, prefixed with `-` when excluded) alongside `toQuery()`.
- Removed stray debug output from the server authentication path.

## Bug Fixes

- **`FilterFactory.rangeDate` now works.** Building a range filter from `java.util.Date` bounds (`FilterFactory.rangeDate(...)`) previously threw `UnsupportedTemporalTypeException` for any populated bound.  It now correctly emits a date range, deriving the calendar date in UTC.
