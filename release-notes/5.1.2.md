## Improvements

- **More-like-this now guards against common terms by default.** Lexical more-like-this queries apply a new `maxDocFreqPct` cap that ignores terms appearing in more than a given percentage of a shard's documents, so boilerplate and other very common tokens no longer dominate term selection. It defaults to `25` and is honored per shard (like `minDocFreq` and IDF scoring). Set it via `MoreLikeThisQuery.setMaxDocFreqPct(int)`: `0` keeps the default of 25, `100` disables the guard entirely, and an explicit absolute `maxDocFreq` still takes precedence when set. Note this changes default behavior: existing lexical more-like-this queries will now drop the most common terms. Pass `maxDocFreqPct(100)` to restore the previous behavior.
- **Vector inputs accept more numeric types.** `MoreLikeThisQuery.addLikeVector(...)` and `VectorTopNQuery` previously accepted only `float[]`. They now offer overloads for `double[]` and any `Collection<? extends Number>`, so callers can pass the vector type they already hold without converting first.

## Bug Fixes

- **Vector query `totalHits` is no longer capped by `amount`.** For `VECTOR` and pure-vector more-like-this queries, `totalHits` was overwritten with the returned result count, which is bounded by the requested `amount`. A search with `vectorTopN: 10000` but `amount: 100` reported `totalHits: 100`, hiding the real KNN candidate pool and making `vectorTopN` look like it was ignored. `totalHits` now reflects the actual candidate pool.
